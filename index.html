Human-in-the-loop n8n — Video 01

「Human-in-the-loop」型 メール自動トリアージ ＆ 返信下書きシステム

n8n / Gmail API / LLM / Slack

▲ メール自動トリアージ デモ動画：メール受信から Slack 通知・返信下書き生成までの自動フローを実演しています。

Problem
自動送信の事故リスクと定型メール対応の工数過多。一件ずつ手動で返信しており、業務のボトルネックとなっていた。
Solution
LLM が感情・文脈・緊急度を分析して自動分類。人間が最終確認・承認（Gmail 下書き保存）する安全な UX を設計し、誤送信ゼロを担保。
Result
担当者は一件ずつ文面を考える時間から解放され、確認と送信の判断だけに集中できます。重要メールは Slack にリアルタイムで届くため、見落としが減り、対応の抜け漏れも防げます。

設計判断：技術的には完全自動返信も実装できました。ただ、実運用で聞こえてくるのは「勝手に送られると怖い」という声の方が多い。特にクレームや感情的な内容が混じるメールでは、AIの判断だけでは不十分な場面があります。そこで、下書き保存＋Slack確認というフローをあえて残しました。担当者が「自分が送った」と確認できることが、現場での安心感と継続利用につながると判断しています。

AIエージェント · パーソナライズ
Personalize AI Dify — Video 02

パーソナライズ AI エージェント（高度な個別最適化のデモ実装）

Dify / 複数データソース統合 / カスタムパラメータ設計

▲ 占いツールデモ動画：個人の特性データを起点に、出力を動的にパーソナライズする AI エージェントの実装例です。

Problem
ユーザーニーズの曖昧さ。一般的な回答では満足度が低く、個人の特性・状況に合わせた出力が求められていた。
Solution
複数データソース（属性・履歴・コンテキスト）を統合し、個人特性に応じて LLM の出力を動的に変化させるパーソナライズ設計を実装。
Result
「この人にはどう答えるか」を担当者が一から考えなくても、状況に合わせた出力が得られます。顧客対応・提案・コーチングなど、画一的な回答では届かない場面での活用が可能な技術基盤を実証しました。

設計判断：「なぜ占いで？」と聞かれることがあります。理由は、パーソナライズの効果が直感的に分かりやすいからです。同じ入力でも人によって出力が変わることを体感してもらうのに、占いは最も誤解が少ないデモ形式でした。技術としては、顧客属性に応じて回答のトーン・内容・深さを動的に変える仕組みで、営業提案やサポート対応など、「同じ質問でも相手によって答えを変えたい」場面に転用できます。小さなデモから現場の用途を想像してもらうことを意図しました。

AI Secretary Dashboard — Video 03

統合型「AI秘書」ダッシュボード：全体概要

LLM API / Calendar API / 独自UI

▲ AI秘書ダッシュボード全体デモ：文章校正・タスク管理・スケジュール調整を一元化した統合UIの操作感を実演しています。

Problem
文章校正・タスク管理・スケジュール調整など日常業務のサポートが複数ツールに分散。ツール間の行き来が発生し、作業効率を下げていた。
Solution
LLM API・Calendar API を組み合わせ、業務サポート機能を一画面に集約した統合ダッシュボードを独自設計。AI が文脈を理解しながら複数タスクを横断的に支援する UX を実装。
Result
複数ツールを行き来する手間がなくなり、「次は何のツールを開くか」と迷う時間が減ります。日々の定型業務をひとつの画面でこなせるため、本来集中したい仕事に向き合いやすくなります。

設計判断：「AI秘書」と名付けましたが、設計で一番気にしたのは「ツールをまた増やすのか」という現場の反応でした。新しいツールが増えるほど、導入初期は混乱しやすい。そのため、既存の Gmail や Calendar API に接続する形をとり、普段使っている情報をそのまま引き込む設計にしています。担当者が「知らないツールを使っている」ではなく「いつもの情報がまとまって見える」と感じられることを優先しました。

AI Secretary UI — Video 04

統合型「AI秘書」ダッシュボード：AI 文章校正機能

LLM API / 独自UI

▲ LLM API と連携し、ビジネスメール等の誤字・文法・文体を自動校正する独自 UI を実装。丁寧度レベルの調整や修正理由の可視化など、実務で使えるUXを追求しています。

Problem
メール文章の敬語・文体チェックに時間がかかり、送信前の確認が属人的に。丁寧すぎる・砕けすぎるといった文体のばらつきも課題だった。
Solution
LLM API を活用し、誤字・文法・文体を自動校正する独自 UI を実装。丁寧度レベルをスライダーで調整でき、修正箇所とその理由も可視化することで納得感のある校正体験を設計。
Result
「この敬語で大丈夫か」と送信前に悩む時間が減ります。修正の理由が可視化されているため、AIの提案を押しつけられた感なく確認でき、担当者が自分で判断して使える設計です。

設計判断：校正ツールを試してもらうと、「なぜこう直されたか分からない」という反応が出やすい。理由が見えない修正は、受け入れにくいのです。担当者が「自分で判断して送った」という感覚を持てないまま使い続けるツールは、だんだん起動されなくなります。修正理由を可視化したのは、そのためです。AIを「答えを出す機械」ではなく、「根拠を示しながら提案する存在」として設計することで、担当者が主体的に使い続けられるUIを目指しました。

AI Secretary UI — Video 05

統合型「AI秘書」ダッシュボード：タスク・スケジュール管理

Calendar API / AI Suggestion

▲ カレンダー API と連携し、複数ツールの行き来をなくすシームレスな UI を構築。AI によるタスク優先度サジェスト機能も実装し、日々の業務効率化を支援します。

Problem
タスク管理・スケジュール確認・優先度判断がそれぞれ別ツールで分散。都度ツールを切り替える手間が集中力を削ぎ、タスクの抜け漏れや締め切り超過につながっていた。
Solution
Calendar API と連携し、タスク一覧・スケジュール・優先度サジェストを一画面に集約。AI が期限・重要度・文脈を考慮して優先順位を提案し、今日やるべきことをひと目で把握できる UI を設計。
Result
「今日は何から手をつければいいか」と迷う時間がなくなります。AIが優先順位を提案するため、タスクの見落としや締め切り超過が減り、業務の流れを自分でコントロールしやすくなります。

設計判断：「AIが自動でタスクを並べ替えてくれる」設計も検討しましたが、実際の現場では「なぜ順番が変わったか分からない」という混乱の方が大きくなりがちです。自分で決めたはずのスケジュールが勝手に動いていると、担当者は不安になる。タスクの優先度は最終的には人間が判断すべきで、AIはその判断材料を整理して渡す役割が適切だと考えました。「提案型」にした理由はそこにあります。

CS Automation Dify — Video 06

CS 対応を自動化する「自己学習型」顧客対応 AI エージェント

Dify / Notion / Slack

▲ FAQ Bot デモ動画：定型問い合わせへの自動応答から、Slack エスカレーション・Notion への回答蓄積による自己学習ループまでを一貫して実演しています。

Problem
「商品の返送」「送料」など定型問い合わせへの手動対応が積み重なり、CS 部門の工数を圧迫。AI が回答できない場合の顧客への案内・担当者への引き継ぎも属人的になっていた。
Solution
Dify × Notion × Slack を連携した実運用特化型ボットを構築。AI が対応困難と判断した場合は顧客へ自動案内しつつ Slack で即時エスカレーション。有人回答は Notion へ自動蓄積し、次回以降 AI が自力回答できる「自己学習ループ」を実装。
Result
担当者は複雑な問い合わせだけに集中でき、同じ質問に繰り返し答える負担から解放されます。有人対応の内容がナレッジとして蓄積されるため、担当者が変わっても対応品質が落ちない体制が整います。

設計判断：FAQボットで現場がよく直面するのは、「最初は使っていたが、いつの間にか担当者が直接答えるようになった」という状況です。原因のひとつは、AIが答えられなかったときの対応が曖昧なこと。担当者が自分で補完するうちに、ボットを経由しなくなる。この仕組みでは、AIが答えられないと判断した瞬間に Slack へエスカレーションし、その有人回答を Notion に蓄積する流れを組み込みました。担当者が変わっても知見が引き継がれ、運用しながら育つ設計にすることで「使われ続けるボット」を目指しています。

HP Auto Build Claude — Video 07

AI によるホームページ自動生成システム

Claude API / HTML/CSS/JS / プロンプトエンジニアリング

▲ ホームページ自動生成デモ：要件をテキストで入力するだけで、デザイン済みの HTML ページが自動生成されるまでの一連のフローを実演しています。

Problem
ホームページ制作には専門知識・時間・費用がかかり、個人事業主や小規模チームにとって大きなハードルだった。「何を伝えたいか」はあっても、それを形にする手段がなかった。
Solution
Claude API を活用し、事業内容・ターゲット・デザインイメージをテキストで入力するだけで、コーディング不要で HTML を自動生成するシステムを実装。要件定義から出力まで一気通貫で完結する設計を採用。
Result
「コードが書けないから頼めない」という壁がなくなります。修正も言葉で指示できるため、エンジニアへの依頼コストをかけずに、自分のペースでホームページを育てていけます。

設計判断：ヒアリングをしていると、「作ってもらったけど、その後触れなくて放置している」という声が出ることがあります。完成品として渡すと、自分のものにならないまま終わってしまう。そのため、まず動くたたき台を早く出して、「ここを直したい」という言葉を引き出すことを優先しました。修正指示を自然言語でできる設計にしているのも、依頼者が主体的に関わり続けられるようにするためです。ツールを渡すより、使いながら育てる体験を渡したいと考えています。

AI Video Gen Remotion — Video 08

自然言語だけで動画を生成：Remotion × Claude Code

Remotion / Claude Code / React / MCP

▲ 動画自動生成デモ：浜松市の天気データをもとに、Claude Code への自然言語指示だけでタイムライン型の天気動画を生成・MP4 書き出しするまでを実演しています。

Problem
動画制作には専用ソフト・編集スキル・長い作業時間が必要だった。「作りたいイメージはあるが、動画にできない」という壁がノンエンジニアにとって高すぎた。
Solution
Remotion（React ベースの動画フレームワーク）と Claude Code を MCP で連携。自然言語の指示だけで AI がコードを生成・修正し、MP4 として書き出すまでを完全自動化。
Result
「動画を作りたいが、ソフトも技術も持っていない」という方でも、伝えたいことを言葉で入力するだけで動画のたたき台が完成します。天気・商品紹介・レポートなど、定期的に発生する定型動画の制作負担を大きく減らせます。

設計判断：動画生成というと技術的な面が注目されがちですが、現場での実用は「毎週同じ形式で作らなければいけない動画」の負担を減らすことにあります。天気・売上・商品紹介など、フォーマットが決まっている定型動画は、毎回編集ソフトを開いて作業するより、入力だけ変えて自動生成できる方が圧倒的に続けやすい。担当者が変わっても同じ品質で出力できることも、運用定着の条件のひとつです。この仕組みはその文脈で設計しています。

Slide Generator Gamma — Video 09

Gamma 用スライド指示文ジェネレーター（提案 A：品質・デザイン特化型）

Claude API / Gamma / プロンプトエンジニアリング / Miro

▲ スライド指示文生成デモ：テーマ・ターゲット・目的を入力するだけで、Gamma にそのまま貼り付けられる構成済みのスライド指示文が自動出力されるまでを実演しています。

Problem
プレゼン資料の作成は「何をどの順番で伝えるか」という構成設計が最も時間のかかる工程。Gamma などの AI スライドツールを使っていても、質の高い指示文を書けないと出力がブレ、結局手直しに時間がかかっていた。
Solution
Claude API を活用し、伝えたい要点を箇条書きで入力するだけで Gamma 用の最適な構成・指示文を自動生成するツールを実装。「人間が考える → AI が構成 → Gamma がデザイン」という半自動フロー（提案 A）を一気通貫で体験できる設計。
Result
「何をどの順番で伝えるか」を考える時間が大幅に減り、内容の吟味と最終調整だけに集中できます。AIスライドツールを使いこなせていない方でも、構成済みの指示文があれば質の高い資料が作りやすくなります。

設計判断：「AIに全部やらせればいい」という方向も検討しましたが、資料は最終的に「人前で話すもの」なので、構成の意図を担当者が理解していないと使いこなせません。AIが自動生成した構成をそのまま貼り付けるだけでは、質問された瞬間に詰まってしまう。Gamma への指示文という形で出力しているのは、担当者が一度内容を確認してから使う機会を設けるためです。また、この指示文自体がドキュメントとして残るため、次の担当者への引き継ぎ資料にもなります。

PowerPoint 自動生成 Claude API — Video 10

テーマ入力だけで .pptx を自動生成：AI スライド自動生成システム

Claude API (Sonnet) / PptxGenJS / Vercel Serverless / HTML/CSS/JS

▲ PowerPoint 自動生成デモ：テーマ・ターゲット・目的を入力するだけで、90秒以内にプロ品質の .pptx ファイルが自動生成・ダウンロードされるまでを実演しています。

Problem
資料作成は「誰が作るか」によって品質がばらつきやすく、構成を考えるだけで多くの時間が費やされます。デザインが得意でない担当者には特に負担が大きく、社内テンプレートがあっても使いこなせないまま属人化しているケースが多くあります。
Solution
テーマ・対象・目的を入力するだけで、AIが「何をどの順番で伝えるか」という構成設計を担い、そのまま .pptx ファイルとして書き出します。デザインや構成のスキルがなくても、誰でも一定の品質で資料を用意できる仕組みです。生成ファイルはそのまま PowerPoint で開いて編集でき、社内テンプレートへの組み込みにも対応しています。
Result
資料作成にかかる時間を大幅に短縮できるだけでなく、「担当者によって資料の質が変わる」という課題が解消されます。チーム全体で共有できる一定品質のベースラインが生まれ、最終確認や調整に集中できる環境をつくります。

設計判断：「90秒で生成できる」より、「誰が作っても同じ品質になる」ことを優先しました。現場でよくある問題は、資料のクオリティが担当者のスキルや経験に依存していることです。得意な人は速く作れるが、苦手な人は時間をかけても仕上がりにばらつきが出る。AIで構成設計を均一化することで、その差を埋めることができます。また、出力を PowerPoint ファイルにしているのは、「最後は自分で調整できる」という安心感を残すためです。完全自動より、担当者が手を入れられる余地を設けた方が、現場では長く使われます。

情報蓄積 · 運用管理
情報蓄積 · 運用管理 Notion / Make — Video 11

情報を「後から追える形」に整える：自動蓄積・共有・運用管理の仕組み

Make / n8n / Notion / Slack / Dify

▲ 情報蓄積・運用管理デモ：会議・問い合わせ・作業履歴などの情報が自動で整理・蓄積され、Notionに記録・Slack通知が流れるまでの流れを実演しています。担当者が変わっても追える状態を目指した設計です。

Problem
情報が担当者のメモや記憶の中に留まり、後から参照できない・引き継げない状況が多くある。AIで作業を自動化しても、その記録がどこにも残らなければ、使い捨てになってしまう。
Solution
Make / n8n で情報の流れを設計し、会議メモ・問い合わせ対応・作業ログなどを自動的に Notion へ蓄積。Slack 通知とセットにすることで「見落とし」を防ぎながら、誰でも同じ場所から情報を参照できる体制をつくる。Dify によるナレッジ検索を組み合わせることで、蓄積された情報を後から引き出しやすくする。
Result
「あの対応どうだったっけ」「誰が確認したのか分からない」という状況が減ります。担当者が変わっても情報は残り、新しいメンバーでも過去の経緯を追えるようになります。一度整えた情報が、繰り返し活用できる資産になります。

設計判断：AIが何かを生成したとしても、それが記録に残らなければ「その場限り」で終わります。担当者が変わる、時間が経つ、別のメンバーが参照したい——そういった場面で情報が追えなくなることは、現場では頻繁に起きます。この仕組みで重視したのは、「後から検索できること」「誰でも同じ場所を見れること」「蓄積が続くこと」の三点です。Notion への自動記録は、その入口に過ぎません。情報が整理された形で残ることが、導入して終わりにしない仕組みの土台になると考えています。

ArchitectureMiro— Arch 01

AIを入れる前に、業務を整理する——人間とAIの役割分担設計図（Miro）

Miro / 業務要件定義 / フロー視覚化

▲ リサーチ・ファクトチェック・回答生成まで、どの工程を誰（人・AI）が担うかを整理した業務設計ドキュメント。ツールを選ぶ前の「構造化」が、現場で使える仕組みの出発点になります。

Problem
「AIを使いたいが、何から始めればいいか分からない」——多くの場合、課題はツールではなく、業務の流れそのものが整理されていないことにあります。構造が曖昧なまま自動化を試みると、途中で止まるか、使われないまま終わります。
Solution
開発より先に、業務を図に描く。図書館でのレファレンス業務と同様に、まず「誰が・何を・どの順番で・どこまで判断するか」を言語化し、人間の確認が必要な工程とAIに委ねられる工程を明確に分けます。この設計図がチームと認識をそろえるための共通言語になります。
Result
「どこをAIに任せ、どこを人間が判断するか」が事前に合意されているため、導入後に運用が止まりません。現場の担当者が自信を持って使い続けられる仕組みは、構造の整理から始まります。

設計の背景：司書として30年間、「相手が何を探しているのか」を引き出すレファレンス業務に携わってきました。曖昧な問いを整理し、必要な情報への道筋を描くこの経験が、AI導入設計の土台になっています。ツールを選ぶ前に業務を構造化することは、図書館で「検索する前に目的を確認する」のと同じ思考です。

Multi-ChannelMake— Arch 03

マルチチャネル・コンテンツ配信パイプライン

Make / LINE / Buffer / OpenAI / 画像生成 AI

▲ Make ワークフロー：LINE 入力から SNS 自動投稿・画像生成まで一気通貫で処理するパイプラインの全体像です。

Problem
各 SNS ごとの投稿工数とトンマナ管理が煩雑。プラットフォームごとに原稿を書き直す二重工数が発生していた。
Solution
LINE からテーマ入力 → 最新情報を自動検索 → X 投稿文・画像を AI 生成 → Router で Buffer / LINE 返信に振り分ける抽象化フローを設計。
Result

設計思想：「単一コンテンツを各 SNS 向けに最適化する抽象化フロー」が設計の核心。ツールに依存しない構造で、将来のチャネル追加にも柔軟に対応できます。

Auto MinutesMake— Arch 04

音声ファイルから自動生成：完全自動化 議事録作成システム

Make / Dropbox / AssemblyAI / OpenAI / Slack

▲ Make ワークフロー：音声ファイルのアップロードをトリガーに、文字起こし→議事録生成→Slack通知まで完全自動で処理するパイプラインの全体像です。

Problem

Result
「会議後の議事録作成」という非生産的な作業時間をゼロに。チームへの共有も即時完了し、意思決定スピードを飛躍的に向上。

設計思想：「ファイルを置くだけ」という最小の操作で全工程が完結するゼロタッチ設計を追求。AI の出力フォーマットもビジネス実務に即した構造に固定し、すぐに使える議事録品質を担保しました。

情報整理・業務設計・AI導入支援｜元図書館司書（30年）・元総務職（5年）

図書館司書として30年間、相談窓口に立ち続けました。来られる方の多くは、自分が何を探しているのかをうまく言葉にできない状態でいらっしゃいます。「なんとなく困っている」「うまく説明できないけど調べたい」——そういう曖昧な言葉を丁寧に受け取り、「本当に必要なもの」を一緒に見つけ、伝わる形へ整えること。それがレファレンス業務の核心であり、今のAI導入支援でも変わらない出発点です。

総務職では、その経験がより現実的な形で試されました。整備されたマニュアルが使われない、引き継ぎがうまく機能しない、担当者が変わると業務が止まる——そういった場面を繰り返し目にしました。仕組みは作るだけでは定着しない。使う人が理解でき、運用ログが残り、誰でも同じように動かせる形になって、はじめて「機能する」のだと実感しました。

AIを活用するようになってからも、この考え方は変わっていません。完全自動化を目指すより、人間が確認できる確認フローを残す。担当者が変わっても回り続け、なぜそう動いているかを説明できる仕組みをつくること。作業時間の短縮だけでなく、ナレッジが蓄積され、次の担当者に引き継げる状態まで設計することが、私の仕事だと思っています。

「何から手をつければいいか分からない」という段階でも、一緒に整理するところから始められます。課題をうまく言葉にできなくても大丈夫です。それを引き出すことが、私の仕事の出発点だからです。まずは話すところから、気軽にご連絡ください。
